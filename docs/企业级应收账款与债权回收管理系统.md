# 项目名称

企业级应收账款与债权回收管理系统

企業向け売掛金管理システム

B2BReceivablesAndCreditManagementSystem

  

# 项目背景

本项目是为一家日本中型商社开发的内部管理系统。

由于该客户拥有大量下游分销商，每月产生大量赊销账单（売掛金）。

为了解决人工核对繁琐、逾期款项追踪困难的问题，开发了此系统来自动化计算迟延损害金并管理督促流程。”

因为日本企业（特别是中小企业）正面临“人手不足”和“电子账簿保存法”改革，急需自动化财务系统。

  

# 项目概要

一个B2B 后台业务支撑系统（Back-office Business System），属于 ERP（企业资源计划） 中的 AR（应收账款）子系统。

这是一个 “数据密集型 + 强规则驱动 + 批处理核心” 的 Web 应用。

核心技术挑战不在于高并发（虽然有月末峰值），而在于 数据的一致性、复杂的业务状态流转（State Machine）以及高精度的数值计算。

前端是台账+图表+数据录入和对账+权限控制。

  

# 用户与权限管理

## 功能描述

基于RBAC模型。

## 基础管理

1. 账号管理：账号均由管理员创建，字段包括姓名、账号、密码和角色、账户状态（禁用/启用）。

2. 认证方式：账号+密码。

## 角色定义:

1. 系统管理员：拥有全系统权限。

2. 营业员：仅能查看自己负责的客户及账单。负责跟进客户回款。

3. 法务专员：可查看数据，尤其是已进入法律程序的账单。

4. 财务专员：拥有核销账单的权限。尽可能开放编辑权限。

  

# 应收账款台账

## 功能描述

核心数据中心，记录所有发出的请求书及回款状态。

## 账单录入

1. 客户信息：企业名称、组织机构代码、联系方式、信用评级（这里参考帝国DataBank / 东京商工Research）、信贷额度、债券合同编号。

2. 账单信息：账单编号、本金金额、账单金额、支付期限、交易日期、利率(%)、逾期开始日、起息日、付款账期。

3. 账单状态：正常、已对账/已结清、部分回款、稟議中、逾期、催款中、诉讼中、坏账核销/已处置(坏账/呆账)

4. 其它信息：担当者、录入时间、备注。

## 列表与查询

1. 支持按“逾期天数（延滞日数）”、“客户等级”、“担当者”进行搜索。 

2. CSV导出：用于经办人员进行对账操作。 

## 账单详情页

1. 实时展示：利息、罚息、本息合计。

  

# 自动入金核销

## 功能概述

监控自己公司银行账户的入金，识别、匹配并核销账单。

1. 银行数据导入：这里用模拟数据，只定义数据结构，参考全银协数据 (Zengin Format)。

2. 定时任务每天核销一遍。

## 智能匹配算法

1. 规则1（完全匹配）：汇款人名义（片假名）+ 金额 完全一致 → 自动通过。

2. 规则2（名义动摇匹配）：去除“株式会社”、“カ）”等前后缀后匹配。

3. 规则3（金额偏差匹配）：允许设定误差范围（如±500日元，通常为银行手续费差异），自动生成“杂损/杂益”会计凭证。 

4. 规则4（合算匹配）：一笔汇款对应多张发票（One-to-Many）的自动凑数算法。

## 消込结果确认

1. 成功匹配的记录自动变为“消込济”。

2. 未匹配记录进入“不明入金列表”，需人工干预。

## 人工核销账单

要做一个左右分栏的界面： 左侧： 银行入金流水（从CSV导入的）。 右侧： 系统的未付账单。

  
  

# 迟延损害金计算

## 功能描述

每日跑批任务计算，基于日本《商法》及合同约定的自动计算逻辑。

## 利息计算规则:

1. 法定利率模式 (默认)：日本《民法》/《商法》规定的法定利率（通常年利 3% 或 6%）。公式：`請求残高 × (法定利率 ÷ 365)`。

2. 契约利率模式 (高风险客户)：合同中约定的惩罚性利率（通常年利 14.6%）。 公式：`請求残高 × (契約年率 14.6% ÷ 365)`。

3. 技术点：需处理日本特有的端数処理（小数点后的金额是四舍五入、切断还是进位，这在日本非常严格）。

## 每日批处理任务(执行时间：每日0点)

1. 处理逻辑：扫描所有状态为“正常/催收中/诉讼中”的账单。

2. 计算当日产生的利息和罚息。

3. 更新数据库中。

4. 如果某笔账单刚刚超过“支付期限”，自动将状态从`請求済`更新为`入金遅延`。

## 自动对账

1. 系统自动读取银行的 **全银协数据 (Zengin Format)**。

2. 将银行流水中的“振込依頼人名 (汇款人名)”与系统里的“請求先 (客户名)”进行模糊匹配。

3. 金额一致且名义一致，自动将状态改为“消込済（已结清）”。

  

# 督促流程

## 功能描述

记录对逾期客户的沟通历史。

## 督促管理

1. 阶段一 (邮件提醒)：超过付款日1天，系统自动发送“入金确认”邮件（礼貌措辞）。

2. 阶段二 (督促状)：超过15天，系统自动生成PDF **“督促状”**，可直接打印或电子发送。

3. 阶段三 (内容证明)：超过60天，生成 **“内容証明郵便”** 模板（具有法律效力的催款函），转入法务流程。

4. 履历管理：记录所有的自动发送记录和客户的回复（Call Log）。

  

# 经营管理报表

## 功能描述

为管理层提供的决策支持数据。

1. 回收予实报表：本月新增应收、本月实际回款、回收率，KPI指标

2. 账龄分析表： 这是财务系统最核心的报表，面试必杀技。展示：30天以内逾期、30-60天逾期、60-90天逾期、90天以上逾期（不良债权）的金额汇总。

  

# 非功能性需求

## 安全性

1. 数据脱敏：银行账号、客户法人个人信息在数据库中加密存储 (AES-256)。

2. 操作日志：所有涉及金额修改的操作必须记录 IP、User ID、Before/After 值。

3. 权限隔离：销售人员只能看到自己负责客户的数据，经理解可以看到全公司数据。

## 性能

1. 并发处理：支持月末高峰期（25日-31日）多人同时上传CSV文件不卡顿。

2. 计算效率：10万条级别的逾期数据利息计算，需在 10分钟内 完成批处理。

## 可用性

1. 系统需部署在 AWS/Azure 云端。

2. 每日定时备份数据库（RDS Snapshot），保留期3年（符合电子账簿保存法要求）。

  
  
  

# 要件定義書 - 売掛金管理システム (RCMS)

  

## 1. プロジェクト概要

本システムは、中堅商社向けのB2B売掛金管理および債権回収支援システムである。

多数の取引先との間で発生する月次の売掛金管理を効率化し、入金消込の自動化、遅延損害金の自動計算、および督促プロセスの可視化を実現することを目的とする。

  

### 1.1 背景

- 取引先数の増加に伴い、手動での入金照合（消込）作業が煩雑化している。

- 支払期限を過ぎた債権に対する遅延損害金の計算が複雑であり、請求漏れが発生している。

- 日本の「電子帳簿保存法」への対応と、バックオフィス業務のDX化が急務となっている。

  

## 2. ユーザー・権限管理

ロールベースアクセス制御（RBAC）を採用し、業務範囲に応じた権限分離を行う。

  

| ロール | 説明 |

| :--- | :--- |

| **管理者 (ADMIN)** | 全機能へのアクセス権限。ユーザー作成、マスタ管理、システム設定。 |

| **財務担当 (FINANCE)** | 入金消込、請求書発行、統計レポートの閲覧。 |

| **営業担当 (SALES)** | 担当顧客の売掛金状況確認、手動での督促ステータス更新。 |

| **法務担当 (LEGAL)** | 法的措置（訴訟）フェーズに移行した債権の閲覧と管理。 |

  

## 3. 機能要件

  

### 3.1 取引先（顧客）管理

- 取引先基本情報（企業名、企業コード、住所、連絡先等）の登録・管理。

- 与信管理（信用格付け、与信限度額）の保持。

- 担当営業者の紐付け。

  

### 3.2 売掛金台帳・請求管理

- **請求書作成**: 請求番号、金額、発行日、支払期限、利率等の登録。

- **ステータス管理**: 以下の状態遷移を管理する。

  - `正常 (NORMAL)`: 支払期限前の状態。

  - `消込済 (PAID)`: 入金が確認され、債権が消滅した状態。

  - `延滞 (OVERDUE)`: 支払期限を超過した状態（システム自動判定）。

  - `督促中 (DUNNING)`: 督促プロセスが開始された状態。

  - `法的措置 (LITIGATION)`: 訴訟等の法的対応に移行した状態。

  

### 3.3 自動入金消込 (Reconciliation)

- **CSVインポート**: 銀行の入金明細データ（全銀フォーマット準拠のシミュレーション）をアップロード。

- **自動マッチング**: 

  - 請求書番号、振込人名義、金額を照合。

  - 完全一致（EXACT）および金額差異（VARIANCE）の判定。

- **消込実行**: マッチング成功時に請求ステータスを「消込済」へ自動更新。

  

### 3.4 遅延損害金計算 (Batch Processing)

- **自動計算**: 毎日0時のバッチ処理により、督促中（DUNNING）の債権に対し遅延損害金を算出。

- **計算ロジック**: `請求残高 × (年利率 / 365) × 経過日数`。

- **端数処理**: 日本の商慣習に基づき、円単位での四捨五入（HALF_UP）を適用。

- **自動ステータス更新**: 支払期限を過ぎた「正常」債権を「延滞」へ自動変更。

  

### 3.5 ダッシュボード・統計

- **主要指標 (KPI)**: 売掛金総額、延滞金額、回収率のリアルタイム表示。

- **債権分析**: ステータス別の債権分布の可視化。

  

## 4. 非機能要件

- **セキュリティ**: Spring Security + JWT による認証。パスワードのハッシュ化保存。

- **データ整合性**: トランザクション管理による入金消込時の整合性確保。

- **拡張性**: 将来的な電子請求書（Peppol等）対応を見据えたデータ構造。

- **インフラ**: AWS（EC2, RDS, S3）へのデプロイを想定。Dockerによるコンテナ化。

- **Frontend**: React 18, TypeScript, Ant Design, Tailwind CSS

- **Database**: MySQL 8.0
