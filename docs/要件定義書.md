# 要件定義書 - 売掛金管理システム (RCMS)

## 1. プロジェクト概要
本システムは、中堅商社向けのB2B売掛金管理および債権回収支援システムである。
多数の取引先との間で発生する月次の売掛金管理を効率化し、入金消込の自動化、遅延損害金の自動計算、および督促プロセスの可視化を実現することを目的とする。

### 1.1 背景
- 取引先数の増加に伴い、手動での入金照合（消込）作業が煩雑化している。
- 支払期限を過ぎた債権に対する遅延損害金の計算が複雑であり、請求漏れが発生している。
- 日本の「電子帳簿保存法」への対応と、バックオフィス業務のDX化が急務となっている。

## 2. ユーザー・権限管理
ロールベースアクセス制御（RBAC）を採用し、業務範囲に応じた権限分離を行う。

| ロール | 説明 |
| :--- | :--- |
| **管理者 (ADMIN)** | 全機能へのアクセス権限。ユーザー作成、マスタ管理、システム設定。 |
| **財務担当 (FINANCE)** | 入金消込、請求書発行、統計レポートの閲覧。 |
| **営業担当 (SALES)** | 担当顧客の売掛金状況確認、手動での督促ステータス更新。 |
| **法務担当 (LEGAL)** | 法的措置（訴訟）フェーズに移行した債権の閲覧と管理。 |

## 3. 機能要件

### 3.1 取引先（顧客）管理
- 取引先基本情報（企業名、企業コード、住所、連絡先等）の登録・管理。
- 与信管理（信用格付け、与信限度額）の保持。
- 担当営業者の紐付け。

### 3.2 売掛金台帳・請求管理
- **請求書作成**: 請求番号、金額、発行日、支払期限、利率等の登録。
- **ステータス管理**: 以下の状態遷移を管理する。
  - `正常 (NORMAL)`: 支払期限前の状態。
  - `消込済 (PAID)`: 入金が確認され、債権が消滅した状態。
  - `延滞 (OVERDUE)`: 支払期限を超過した状態（システム自動判定）。
  - `督促中 (DUNNING)`: 督促プロセスが開始された状態。
  - `法的措置 (LITIGATION)`: 訴訟等の法的対応に移行した状態。

### 3.3 自動入金消込 (Reconciliation)
- **CSVインポート**: 銀行の入金明細データ（全銀フォーマット準拠のシミュレーション）をアップロード。
- **自動マッチング**: 
  - 請求書番号、振込人名義、金額を照合。
  - 完全一致（EXACT）および金額差異（VARIANCE）の判定。
- **消込実行**: マッチング成功時に請求ステータスを「消込済」へ自動更新。

### 3.4 遅延損害金計算 (Batch Processing)
- **自動計算**: 毎日0時のバッチ処理により、督促中（DUNNING）の債権に対し遅延損害金を算出。
- **計算ロジック**: `請求残高 × (年利率 / 365) × 経過日数`。
- **端数処理**: 日本の商慣習に基づき、円単位での四捨五入（HALF_UP）を適用。
- **自動ステータス更新**: 支払期限を過ぎた「正常」債権を「延滞」へ自動変更。

### 3.5 ダッシュボード・統計
- **主要指標 (KPI)**: 売掛金総額、延滞金額、回収率のリアルタイム表示。
- **債権分析**: ステータス別の債権分布の可視化。

## 4. 非機能要件
- **セキュリティ**: Spring Security + JWT による認証。パスワードのハッシュ化保存。
- **データ整合性**: トランザクション管理による入金消込時の整合性確保。
- **拡張性**: 将来的な電子請求書（Peppol等）対応を見据えたデータ構造。
- **インフラ**: AWS（EC2, RDS, S3）へのデプロイを想定。Dockerによるコンテナ化。

## 5. 技術スタック
- **Backend**: Java 17, Spring Boot 3.x, Spring Data JPA, Spring Security
- **Frontend**: React 18, TypeScript, Ant Design, Tailwind CSS
- **Database**: MySQL 8.0
- **DevOps**: Docker, GitHub Actions
